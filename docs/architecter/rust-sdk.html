IM Core SDK 是否适合 App 与 CLI 共用 Rust SDK：调研报告

1. 调研计划与执行范围

本次调研按只读方式执行，没有修改任何文件。计划如下：

  1. 梳理当前 Awiki Me App 新增的 lib/src/im_core API、DTO、事件、错误与 reserved 接口。
  2. 梳理 ../awiki-cli 中 IM 相关命令和服务能力，包括 msg、attachment、group、msg.secure、 group.e2ee、runtime listener / host notify。
  3. 按能力维度对比 App IM Core 与 CLI IM 能力，判断是否可以抽象成同一套 Rust SDK。
  4. 输出结论、差距、风险和建议架构。
总判断： 当前 IM Core API 的能力面 基本适合作为 App 与 CLI 共用 Rust SDK 的语义蓝图。 但当前 Dart API 不应被直接视为 Rust ABI；更合理的结构是： Rust SDK 定义 canonical core API，然后分别提供 Dart adapter 和 Go/CLI adapter。

2. 总体结论

问题 结论 置信度
当前 IM Core API 是否覆盖 App 和 CLI 共同需要的核心 IM 能力？ 基本覆盖 direct/group 消息、会话/历史、附件、群组、实时事件、本地状态、outbox、reserved secure/E2EE。 高
是否可以让 Awiki CLI 和 Awiki Me App 底层共用一套 Rust SDK？ 可以，但应通过 Rust canonical core + 双端 adapter，而不是让 CLI 直接依赖 Dart facade。 高

当前 Dart IM Core 是否可以直接成为 Rust SDK ABI？ 不建议，Dart 中有 Stream、DateTime、Uri、Map<String,Object?> 等 语言/平台相关类型。 高

5. 是否能“一套 Rust SDK 同时给 CLI 和 App 用”

答案：可以，但要分清三层接口。

Rust awiki_im_core

  ├─ canonical IM engine
  │   ├─ protocol / RPC / signing boundary
  │   ├─ direct & group message send/list/read
  │   ├─ attachment upload/download
  │   ├─ group lifecycle
  │   ├─ realtime event stream
  │   ├─ local IM store / unread / outbox
  │   └─ direct secure / group E2EE / migration
  │
  ├─ Dart binding adapter
  │   └─ implements AwikiImClient for Awiki Me App
  │
  └─ Go/CLI binding adapter
  └─ maps Rust DTOs to awiki-cli CommandResult / json / pretty / table

当前 lib/src/im_core 的设计适合做 Dart binding facade。 它不是最终 Rust ABI，但可以反推 Rust SDK 需要提供的能力。

6. 主要差距与需要补强的接口点

  1. 需要 Rust canonical API 设计层。 当前 Dart API 使用 Stream、DateTime、Uri、Map<String,Object?>， 这些不适合作为 Rust/Go/Dart 共同 ABI。Rust 层应定义语言无关 DTO，再由 Dart/Go adapter 转换。
  2. CLI inbox 语义需要明确。 CLI 的 msg.inbox 支持 scope、with、group、unread、mark-read。 当前 IM Core 没有同名 API，可以组合实现，但若要让 CLI adapter 简洁，建议 Rust SDK 增加 inbox/listRecent。
  3. Secure/E2EE 的阶段语义要重新定级。 App Phase 1 中 directSecure/groupE2ee 是 reserved；CLI 目前已有实现面。 如果共用 Rust SDK 的目标包含替换 CLI secure/E2EE，则这些能力不能长期 reserved。
  4. CommandResult 不应进入 Rust Core。 CLI 的 CommandResult{Data, Summary, Warnings} 是命令行输出模型。 Rust SDK 应返回领域 DTO / error / warning；CLI adapter 再渲染 json/pretty/table。
  5. runtime listener/host-notify 应保持在外层。 Rust IM Core 只负责连接和事件；CLI listener service、host-notify、App push/UI notification 由各自 runtime 层处理。
  6. 文件/附件 IO 策略需要跨端统一。 App 可能传 bytes/localPath；CLI 主要传 file path/output path。 Rust SDK 需要明确支持 byte stream、local path、download output 的跨平台边界。

7. 推荐的下一步接口调整

优先级 建议 原因
P0 定义 Rust canonical IM Core API，不直接复用 Dart API 作为 ABI。 避免 Dart/Flutter 类型泄漏到 CLI/Rust 层。
P0 明确 adapter 边界：Dart adapter 实现 AwikiImClient；Go adapter 服务 CLI command handlers。 保证 App/CLI 共享核心但保留各自输出/UI/runtime 逻辑。
P1 补一个 Rust SDK InboxApi 或定义 CLI inbox 如何组合 sync + messages.list + markRead。 CLI msg.inbox 是核心命令，语义比 App conversation list 更偏消息流。
P1 重新标定 directSecure/groupE2ee：App Phase 1 reserved，Rust SDK shared target 可以是 v2/v3 required。 CLI 已有 secure/E2EE 命令面，共用 SDK 最终应承接这些能力。
P1 为 warnings/errors 定义跨端结构。 CLI 当前依赖 warnings；App 依赖 event/error DTO；Rust SDK 应统一错误/警告模型。
P2 把 runtime listener/host-notify 明确排除在 IM Core 外，只消费 event stream。 防止 CLI runtime 和 App notification 逻辑污染 SDK core。

8. 最终判断

当前 IM Core SDK 接口设计方向是对的。 它已经覆盖 App 与 CLI 共同需要的大部分 IM 能力，尤其是消息、附件、群组、实时事件、本地状态、outbox、reserved secure/E2EE。

但如果目标是“Awiki CLI 和 Awiki Me App 底下都用同一套 Rust SDK”，下一步不应该直接实现 Dart API 的 Rust 版本， 而应该先定义一个 语言无关的 Rust canonical IM Core API， 然后让 Dart 和 Go 各自适配。

推荐结论： 可以共用一套 Rust SDK ，当前接口可以作为能力蓝图； 但需要补充 Rust canonical API / CLI inbox / secure-E2EE 阶段 / adapter 边界。
